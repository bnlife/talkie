**结论**: 不改。Pinia 跨 store 读取是正常用法，依赖清晰且单向。如果按"参数传入"重构会增加不必要的中间层。

# 2. perform_search 硬编码搜索 MCP 服务器 ID（暂不改）

**文件**: `src-tauri/src/commands/chat.rs` L320-342

**现状**:
- 匹配规则: `server_id == "brave-search" || "duckduckgo" || "bocha-search" || "local:bocha-search" || server_id.contains("search")`
- 工具名映射: bocha → `bocha_search`，其他 → `search`

**风险**: 用户装了名字不含 "search" 的搜索 MCP（如 `web-lookup`）不会被识别；工具名不是 `search` 的搜索 MCP 调用会失败。

**触发条件**: 等有用户反馈装了搜索服务但不被识别时再改。

**改造方向**: `mcp_instances` 表加 `capability` 字段（如 `"search"`），或 MCP 注册表加 `tags: ["search"]` + `default_tool_name`，按 capability 匹配替代硬编码 ID。

# 4. thinking 模式硬编码开启（待 provider 页面改造）

**文件**: `src-tauri/src/llm.rs` L57-59

**现状**: 请求体中 `thinking: { type: "enabled" }` 对所有模型无条件开启。当前只用 MiMo 和 DeepSeek，两者都支持。但未来如果接入不支持 thinking 的模型（如 GPT-4o），会返回错误或忽略该参数。

**触发条件**: 开 provider 页面让用户自选模型时。

**改造方向**:
1. `ModelProvider` 新增 `supports_thinking: bool` 字段（或按模型名自动检测）
2. `llm::stream_chat` 根据该字段决定是否发送 `thinking` 参数
3. provider 页面增加"是否支持 Thinking"开关或自动检测提示
