# 核心长文件拆分分析报告

除了已讨论的 `store.rs` 和 `stores.spec.ts`，项目中还有几个超过 400 行的文件。以下是针对这些文件的拆分必要性分析及专业方案。

---

## 1. `McpView.vue` (517 行) —— **必须拆分**

### 分析：
该文件目前是一个“巨型组件”，它同时承担了：
1. **窗口控制**（最小化/最大化/关闭）。
2. **侧边栏导航**（分类切换、已安装实例列表）。
3. **市场视图**（服务卡片展示）。
4. **实例详情**（状态显示、配置查看、测试连接）。
5. **多个弹窗逻辑**（安装配置弹窗、自定义服务弹窗）。

### 建议方案：
按照“容器-组件”模式进行拆分：
*   **`McpView.vue`**：作为父容器，仅保留状态管理（Pinia 交互）和基本布局。
*   **`McpSidebar.vue`**：提取侧边栏逻辑（分类列表、已安装列表、搜索框）。
*   **`McpMarketGrid.vue`**：提取市场服务卡片列表逻辑。
*   **`McpInstanceDetail.vue`**：提取已安装实例的详情展示与操作逻辑。
*   **`McpInstallDialog.vue`**：将复杂的安装表单和校验逻辑封装为独立组件。

**专业收益**：UI 逻辑解耦，每个组件的代码量将下降到 100-200 行，极大提升模板的可读性。

---

## 2. `src-tauri/src/commands/chat.rs` (587 行) —— **建议拆分**

### 分析：
这是后端最核心的命令文件，目前包含了：
1. **上下文构建** (`gather_context`)。
2. **LLM 配置解析** (`resolve_llm_config`)。
3. **流式执行引擎** (`execute_stream`)。
4. **搜索集成逻辑** (`perform_search`)。
5. **前端交互命令** (`send_message`, `regenerate_message` 等)。

### 建议方案：
将编排逻辑与命令接口分离：
*   **`src-tauri/src/chat/engine.rs`**：提取 `execute_stream`、`gather_context` 和 `finalize_response`。这部分是纯业务逻辑，不直接依赖 Tauri Command。
*   **`src-tauri/src/chat/search.rs`**：提取 `perform_search` 及其相关的搜索结果解析逻辑。
*   **`src-tauri/src/commands/chat.rs`**：仅保留作为入口的 Tauri `#[tauri::command]` 函数，调用上述引擎。

**专业收益**：实现“薄控制器，厚 Service”的架构，方便对核心生成逻辑进行独立的单元测试（不需要 Mock Tauri 环境）。

---

## 3. `mcp-plan.md` (500 行) —— **无需拆分，但需管理**

### 分析：
这是一个文档文件，记录了 MCP 的实现计划。

### 建议方案：
*   **无需拆分**：文档文件通常不需要像代码一样拆分。
*   **建议**：如果计划已完成，可以将其移动到 `docs/archive/` 目录下，或者将其中的技术决策提取到 `docs/architecture.md` 中。

---

## 总结建议

| 文件 | 优先级 | 拆分策略 |
| :--- | :--- | :--- |
| **`McpView.vue`** | **极高** | 拆分为 4-5 个功能组件 |
| **`chat.rs`** | **高** | 提取业务引擎 (Engine) 和搜索模块 (Search) |
| **`mcp-plan.md`** | 低 | 归档或合并至架构文档 |

**重构原则**：
1.  **UI 拆分看视觉边界**：一个视觉块对应一个组件。
2.  **后端拆分看职责边界**：IO 交互（Command）与 业务逻辑（Engine）分离。
